# UI Audit — 2026-05-07

**Scope:** `web/operator-ui/src/app/{pages,components}` の全画面を歩いて、 「これは直す価値がある」 と判断した点を一個ずつ punch list にしたもの。 「綺麗にしよう」 の漠然から抜け出すための叩き台。

**Reviewer の立場:** Claude が junior Rust partner として、 UX (= 操作主が "これ何?" "次どこ押す?" "何が起きた?" で迷う回数) の観点で読んだ判断。 美的判断 (色、 余白) はほぼ含んでいません。

## 凡例

- **カテゴリ**:
  - **D** = density (1 画面に詰め込み過ぎ)
  - **C** = click cost (常用操作のクリック数 / 探す手間)
  - **N** = naming / inconsistency (用語ぶれ、 jargon 露出)
  - **F** = state feedback (操作結果が見えにくい / 何が起きたか分からない)
  - **L** = layout (整列、 はみ出し、 narrow window で崩れる)
  - **A** = accessibility (キーボード、 コントラスト、 aria)
- **規模**: **S** < 1h、 **M** 1–4h、 **L** > 4h
- **ステータス**: 各項目は punch list の素案。 「○ 直す / × 放置 / △ 後で」 の判定を上に書き込む形でレビューしてください。

## 全体 (cross-cutting)

### X-1 [N/F] [S] スナックバーで `MatSnackBar.open()` 直接呼び出しが残存していないか念押し
直す
R-P4 (`111b9fd`) で 12 site を `NotificationService` に統一済。 ただし `monitor.component.ts:197` で `private readonly snackBar = inject(MatSnackBar)` が残っている (移行漏れ)。 加えて `control-panel.component.ts:8` も `MatSnackBar` を import している。 追加レビューで `grep -n "MatSnackBar\b" web/operator-ui/src` を流して全部 NotificationService 経由に。

### X-2 [N] [S] "Online" バッジが何の online か分からない
問題ないので放置
ヘッダ右端の `.status-indicator` は `operator.status()` (= バックエンド到達性 1 fetch ぶん) を出している。 だが画面には Component Status カードに各コンポーネントの online/offline がもう一度出ている。 操作主は "Online って何が online?" と一瞬考える。 → "Backend" ラベル併記、 もしくは tooltip で "Operator API to backend" を明示。

### X-3 [D] [S] Idle 中に "0 events | 0 eve/s" を出さない
Idle と明示されているので問題ない
ヘッダの header-stats は run 中以外は意味がない数字。 Idle / Configured 中はサプレスして "Ready" / "Configured" だけ出す方が状態認知が早い。

### X-4 [N] [M] "warn" カラー (赤) が Stop と Cancel/Reset の両方に使われている
試してみましょう
Material の `color="warn"` は破壊的アクションの記号。 Stop は run-stop で正しい使い方だが、 control-panel:84 の Cancel-edit ボタン、 emulator-settings 系の Reset、 Run timer Stop、 Settings の Reset (これは ng model のローカルリセット) など 5 箇所で同じ赤を共有している。 操作主は "赤 = やばいやつ" の脳内マップが薄まる。 → Reset 系は warn を外す or `mat-stroked-button` に。

### X-5 [F] [S] Apply 失敗時に system_state がどう変わったか UI に出ていない
直すというか実装お願いします
R-P4 後の firmware mismatch fix (`91e3505`) で `auto_config_failed = true` で Arm を blocking するようになった。 ところが UI 側は赤 snackbar を出すだけで、 「Configure ボタンが再び push 必要」 という guidance が無い。 → Apply 失敗時に Configure ボタンの近くに "Apply failed — fix config and Configure again" の inline alert を出す。 (snackbar はすぐ消える前提の補助通知)

### X-6 [N] [S] FW 名称が UI で複数表記揺れ ("PHA" vs "PHA1" 等)
後回し
過去の修正で `PHA → PHA1` 統一は済んでいるが、 firmware-badge の class 名が `.psd2 / .pha2 / .amax / .x743std` のように小文字で混在 (digitizer-settings:70)、 一方 enum 値は `'PSD2'/'PHA2'/'AMax'/'X743Std'`。 視認上は OK だが、 検索/grep が辛い。 → CSS 側を `.fw-psd2` など接頭辞付きに揃えると安心。

### X-7 [A] [M] キーボードショートカットが Configure/Apply/Start/Stop に無い
気持ちはわかるが、私のユーザーにはショートカットの概念がある人が少ないし二時間に一回くらいしか本番は操作しないので問題ない。本番前のチューニングでもパラメータをアプライするだけで、大半の場合は Enter で Apply されるのでそこまで問題ない。
最頻常用アクション 4 個に `Ctrl+Enter` (Apply)、 `F5` (Configure) みたいな割り当てが無い。 操作主は毎回マウスでクリック。 操作頻度を考えるとコスパ良い改善。 既に Tune Up tab の `tabindex="0" (keydown.enter)` で Apply を Enter にしているので前例あり。

## Control 画面

### CT-1 [D] [M] Component Status カードの Pipeline ヘッダが折り畳めるように見えて折り畳めない
pipeline は将来にわたってそんなに増えることはない（せいぜいEBを足すくらい）のでアイコンを外すで
`status-panel.component.ts:73` で `<mat-icon>expand_more</mat-icon>` を出しているが `cursor: default` で click handler 無し。 Readers 側は折り畳めるのでアイコンが同じだと "あれ折り畳めるんだっけ" と毎回試す。 → アイコンを外すか、 折り畳めるようにする。

### CT-2 [D] [S] コンポーネント行の inline metrics が読みにくい
問題ないので放置
`· 1.2M · 5k eve/s · 50 MB · LOSS: 0` のように中黒区切りで 4 メトリクスが横一列。 LOSS が 0 のときは出さない (>0 のときだけ赤で出す) になっているのは○。 ただし events / rate / bytes は常時表示。 多数 digitizer 環境で縦長ペインが詰まる。 → 数値だけにしてラベルを tooltip に追い出す、 もしくは "1.2M ev · 5k/s · 50MB" の "ev" "/s" 単位を統一して読みやすく。

### CT-3 [N/D] [M] Control Panel の Run Number に edit/confirm/cancel UI を常時出す必要があるか
やってください
`control-panel.component.ts:60-89` の鉛筆ボタン → check / × ボタン flow は安全策として正しいが、 通常運用では Auto なのでほぼ触らない。 → 折り畳んで "Override Run #" の小さい link 1 個 (default は collapsed) に。 普通の人はそもそも気付かなくていい。

### CT-4 [L] [S] 4 ボタン (Configure / Reset / Start / Stop) のうち Reset だけ色が無い
現状はどれも別の色にされてませんよ
`control-panel.component.ts:111-130`。 `Configure` は primary、 `Start` は accent、 `Stop` は warn、 `Reset` だけ color 指定無し → 灰色。 視覚バランスとして奇妙。 加えて Reset の意味が不明 (これは run state machine の Reset コマンド = "全部 Idle に戻す")。 → 機能名を "Force Reset" に変えて warn 色 + 確認ダイアログ、 通常運用ではあまり見えない位置に。

### CT-5 [D] [M] Comment と Run Notes が同じ画面内に共存している
やらないでください。実験中にComment は必ず見えないとおかしい
Run 開始前は "Comment (for Start)" のテキストエリアが見え、 Running 中はそれが disabled になり下に "Run Notes" セクションが追加で展開される。 結果、 Running 中は (使えない) Comment + (使える) Notes で 2 つのテキスト入力が縦に並ぶ。 → Running になったら Comment を畳む (subtle "Initial comment: ..." の 1 行 readonly に格下げ)。

### CT-6 [D] [M] Timer カードが Control 系から離れて独立している
試しにやってください
Timer (右下) は Control Panel と機能的に紐付いている (Start with Timer は内部で controlPanel の displayRunNumber を読む) のに UI 上は別カード。 操作主からは "Start Timer" と "Start" の関係が分かりにくい。 → 1 ヶ所に統合 (Start ボタン横に "with timer ☐" + duration input)、 もしくは Timer カードに Start/Stop ボタンを置かない (混乱の元)。

## Settings / Digitizers 画面 (最大の負債)

### DG-1 [D] [L] ヘッダ行が 7 要素で narrow window で必ず折れる
やってください
Select Digitizer + Name + firmware-badge + S/N + spacer + Detect + Reset + (Reset AMax) + Apply。 1280px 以下で改行する。 → ヘッダは "Select Digitizer + firmware-badge + S/N + Apply" の 4 要素に絞る、 残り (Name 編集 / Detect / Reset / Reset AMax) は kebab メニュー or 小さい "Advanced" panel に。

### DG-2 [N/F] [S] "Detect" "Reset" "Reset AMax" "Apply" が同じ視覚重みで並んでいる
やってください
Apply は最重要、 Detect は初期セットアップ時のみ、 Reset は edit cancel、 Reset AMax は AMax 開発者向け診断。 全部 `mat-button` か `mat-raised-button` で大きさ近い。 → Apply のみ raised-primary、 残り 3 個を `mat-icon-button` の小型化 + tooltip。 視線が Apply に集まる。

### DG-3 [N] [M] "Reset" の意味が 3 通りある
直してください
(a) digitizer-settings:88 `resetConfig()` = ng model をサーバ値に戻す、 (b) AMax の `Reset AMax` = FW 開発者デフォルト復元、 (c) Control Panel の `Reset` = state machine reset。 全部 "Reset" 表示。 → (a) を "Discard changes"、 (b) は "Restore FW Defaults"、 (c) を "Force Reset" などに rename。

### DG-4 [D] [L] Board タブが 1 枚の長い縦スクロールカード
x743は特殊なので現状これでいい
Clock & Sync / Trigger & I/O / Test Pulse / Board Veto / Data Acquisition の 5-7 セクション (FW で変動) が `<mat-divider>` で区切られて縦に並ぶ。 PHA2 で約 20 input field、 X743 で約 25 input field + advanced register editor。 操作主は "Clock 設定どこだっけ" でスクロール往復。 → 各セクションを `<mat-expansion-panel>` (collapsible) にして 1 画面に headlines を出す、 もしくは Board タブ内で更にサブ tab。

### DG-5 [N] [S] Board タブの enum 値が CAEN jargon そのまま
フロントとバックでちゃんと方言ではなく共通の名前でやり取りしてるか確認お願いします。その上でなおisてください
`SINlevel`, `SINedge`, `EncodedClkIn`, `LVDS`, `P0`, `ITLA_AND_ITLB`, `START_MODE_S_IN`, `FPIOTYPE_NIM` 等。 操作主が初見だと CAEN UM を引かないと分からない。 → "S-IN (level)" "S-IN (edge)" "External Clock (encoded)" のような説明的ラベル + `matTooltip` で原語を保持。

### DG-6 [N] [S] PSD2/PHA2 と PSD1/PHA1 で Start Source の enum 値表記が違う
お願いします
PSD2/PHA2: `EncodedClkIn` `SINlevel` `SINedge` `SWcmd` `LVDS` `P0`、 PSD1/PHA1: `START_MODE_SW` `START_MODE_S_IN` `START_MODE_FIRST_TRG`。 後者だけプレフィックスが入っているので並べて見ると違和感。 → DG-5 と同じ仕事で揃える。

### DG-7 [F/N] [S] "Apply (Runtime)" のラベルが状態依存
それがいいですね
Running 中だけ Apply ボタンが "Apply (Runtime)" になる (digitizer-settings:111)。 tooltip で "Only SetInRun parameters will be applied"。 操作主は label 変化で気付くが、 tooltip まで読まないと "残りのパラメータどうなる?" が分からない。 → Apply ボタンの下に inline ヒント "11/43 params can be applied at runtime — full apply needs Stop+Configure" を出す (running 中のみ)。

### DG-8 [D] [M] Coincidence タブが X743 で N/A 専用画面になる
可能ならお願いします
X743 では Coincidence の中身が `"Coincidence settings are not applicable for V1743 Standard mode."` 1 行。 タブ自体が無駄。 → FW 別に tab を出し分け、 X743 で Coincidence を消す (動的 tab 構成)。

### DG-9 [D] [M] Energy タブが FW で別物すぎる
x743は特殊でまだ使うかどうかのテスト中なので変更は不要
X743: 4 input (Energy Source / Baseline Samples / Energy Scale / Energy Offset) のボードレベル設定。 PSD2/PHA2/PSD1/PHA1: チャンネルテーブル (per-channel parameters)。 同じ "Energy" タブだが粒度がボード vs チャンネルで根本的に違う。 → X743 のは Board タブの最下部 "Post-processing" セクションに移し、 Energy タブを X743 では非表示。

### DG-10 [N] [S] firmware-badge の表示が冗長
サイズ的に可能ならお願いします
ヘッダに `PHA2` のバッジ + S/N + その下のタブで Board / Input / Trigger / Energy / Coincidence / Waveform。 タブの内容が FW で変わるので、 タブラベル側に "Board (PHA2)" のような hint があると "あ、 PHA2 用の Board を見ているのか" が一目でわかる (DG-4 の expansion panel 化と組み合わせると更に明瞭)。

### DG-11 [A] [S] Board タブの form-grid が 2 列だが、 セクション内 grid が散在
今ので問題なし
DG-4 の長スクロール問題と関連。 グリッドが Section ごとに 2 列で揃っているが、 セクション間の余白がほぼ無いので "ここまでが Trigger & I/O" の境界が divider の細い線だけ。 → セクションタイトルのフォントサイズを 1px 上げる + divider の上下マージンを 24px に増やす、 で読み取りグリッドの目を作る。

## Waveform 画面

### WF-1 [D] [L] Tune Up モードのツールバーが 1 行に 15+ コントロール
問題ないので後回し
Tune Up badge + 名前 + Channel select + 6 probe checkbox + Run/Single button + status badge + Acc toggle + N input + count + clear + spacer + Stop。 1920px でも横ぎりぎり。 操作主の最頻利用は Channel 切替と probe on/off。 → Probe を別行 or 折り畳み、 Acc 系を gear/menu に追い出す。

### WF-2 [N] [M] probe checkbox が常に 6 個 (A0/A1/D0..3) 出る
今のままでいいと思います。結局は知ってる人間しかここは触らないので

### WF-3 [F] [S] "ARMED" / "HOLD" バッジが目立たない
やってください

### WF-4 [D] [L] Tune Up bottom row の param table の "All" モードが 5 カテゴリの grid
確かにAll は全チャンネルの初期状態を作るのに使うくらいですが、Settings と Tune Up Mode を行き来するのがだるいので今の実装です。議論しましょう

### WF-5 [D/N] [M] Normal モードと Tune Up モードが同じ URL `/waveform`
基本的に waveform を必要とするときは tune up 目的なので、ほぼ問題ない。議論しましょう
`if (isTuneUp())` で template 全体を分岐 (1700 行のうち半分はこの switch)。 操作主からは「同じ画面の中で違うことしてる」ので浅い理解だと混乱する。 → 別 route `/tuneup` にする、 もしくは Tune Up 画面を Settings/Digitizers の下に持っていく (Tune Up は config 編集ループなので、 概念的には Settings に近い)。

## Runs 画面

### RN-1 [D] [M] expand 行のコンテンツが多すぎる
link で別ページにラン詳細が出るのはいいですね。実際一覧では開始、終了時刻（その時間差）とイベントレート、ロストトリガ数くらい見れたらよくデジタイザの設定は詳細ページでいいですね。ちなみにデジタイザのリストだとかは id の順にしてください。現状はランダムに見えてわかりにくいです。
`runs.component.ts:104-222`。 1 run を expand すると、 9 メタ field + 全 digitizer 設定の Board / Channel Defaults / Channel Overrides の KV テーブル + 内部 tab (digitizer 切替)。 1 run の縦長が 1000px 超。 → expand は run のサマリ + "View full config →" link で別ページ or modal、 もしくは expand 内で sub-section も collapsible に。

### RN-2 [N] [S] Trigger Loss の % 表示の分母が事後計算
バックで計算してるならそれを使ってください
`(loss / (events + loss) * 100).toFixed(3)`。 backend の `trigger_loss_rate` を信用せず再計算している。 → `run.stats.trigger_loss_rate` を出すだけにすると (a) 一貫、 (b) 速い、 (c) 計算ミス防げる。

### RN-3 [F] [M] config snapshot が無い run と "あるけど empty" の区別が無い
お願いします
`@if (configSnapshot() === null || configSnapshot()!.length === 0)` で "No configuration snapshot available for this run." を出すが、 `null` (= まだ load していない) と `[]` (= load 済だが backend に snapshot 無し) の区別が無い。 → null のときは spinner or "Loading..."、 [] のときは "Snapshot wasn't recorded for this run (recorded since 2026-02-19)" のような absolute hint。

### RN-4 [N] [S] Master/Slave バッジが Master だけ
確かにこれは Master/Slave を元に Start Mode を自動割り当てする予定でしたが、現在は Settings でそれらを明示的に設定するので不要です。そして Master/Slave が政治的に正しくないという全てのクソッタレどもに膝を屈する意味でもなくしましょう。
`@if (cfg.is_master) { <span class="master-badge">Master</span> }`。 そもそも `is_master` は廃止予定 (memory: "is_master 削除: TOML/Operator config から冗長な is_master を除去" 2026-03-12)。 backend snapshot にもう書かれていない可能性。 → backend が `is_master` を出していないか確認、 出していなければバッジ自体削除。
